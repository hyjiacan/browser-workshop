# 日志系统

bws 采用双输出日志系统，文件和控制台独立控制级别，既能保留完整的调试信息，又能保持控制台输出的简洁。

## 双输出日志系统

bws 的日志系统同时输出到两个地方：文件和控制台，两者的日志级别独立控制。

### 输出对比

| 输出 | 默认级别 | 内容特点 | 位置 |
|------|----------|----------|------|
| 文件日志 | `DEBUG` | 时间戳 + 级别 + 源码位置 + 详细信息 | `logs/bws.log` |
| 控制台 | `INFO` | 简洁的用户可见信息 | stderr（标准错误输出） |

### 设计理念

- **文件日志**：完整记录所有操作，用于问题排查和事后分析
- **控制台**：只输出用户关心的信息，保持界面整洁
- **独立配置**：两者级别独立，修改控制台级别不影响文件日志的完整性

## 日志级别说明

bws 支持 6 个日志级别，从低到高依次为：

| 级别 | 说明 | 示例场景 | 文件默认 | 控制台默认 |
|------|------|----------|----------|-----------|
| `trace` | 最详细的跟踪信息 | 函数调用、变量值 | - | - |
| `debug` | 调试信息 | HTTP 请求、文件操作路径 | ✓ | - |
| `info` | 一般操作信息 | 安装开始、下载进度 | ✓ | ✓ |
| `warn` | 警告信息 | 不影响运行的异常情况 | ✓ | ✓ |
| `error` | 错误信息 | 操作失败、网络错误 | ✓ | ✓ |
| `fatal` | 致命错误 | 程序崩溃、无法恢复 | ✓ | ✓ |

### 各级别输出内容

#### trace

最详细的日志级别，通常用于深度调试：

- 函数进入和退出
- 变量值变化
- 循环迭代信息
- 底层库调用

#### debug

调试信息，用于排查问题：

- HTTP 请求和响应详情
- 文件读写路径
- 配置加载过程
- 版本匹配过程
- 解析和识别过程

#### info

一般操作信息，用户通常会关心：

- 安装开始和完成
- 下载进度
- 导入结果统计
- 配置变更
- 服务启动和停止

#### warn

警告信息，表示有异常但不影响继续运行：

- 文件名无法识别（但继续处理其他文件）
- 网络重试
- 缓存过期
- 配置项使用默认值

#### error

错误信息，表示某项操作失败：

- 下载失败
- 安装失败
- 文件损坏
- 配置无效
- 权限不足

#### fatal

致命错误，程序无法继续运行：

- 数据目录无法创建
- 配置文件损坏且无法恢复
- 严重的内部错误

## 日志文件位置

### 便携模式

```
bws-data/
└── logs/
    └── bws.log
```

### 传统模式

如果设置了 `BWS_HOME` 或 `data-dir`：

```
$data-dir/logs/bws.log
```

如果使用用户主目录：

```
~/.bws/logs/bws.log
```

### 日志文件格式

每行日志包含以下信息：

```
时间戳  级别  源码位置  消息内容
```

示例：

```
2024-01-15T10:30:00.123Z  INFO  install/install.go:45  开始安装 chrome 120.0.6099.109
2024-01-15T10:30:00.456Z  DEBUG download/download.go:78  下载地址: https://dl.google.com/chrome/...
2024-01-15T10:30:05.789Z  INFO  download/download.go:120  下载进度: 50% (50MB/100MB)
2024-01-15T10:30:10.012Z  INFO  install/install.go:120  安装完成
```

### 日志轮转

日志文件会自动轮转，防止单个文件过大：

- 单个日志文件达到一定大小时自动创建新文件
- 保留一定数量的历史日志文件
- 旧日志文件会被自动清理

## 修改控制台日志级别

使用 `bws config` 命令修改控制台日志级别。

### 设置日志级别

```bash
bws config set log-level debug
```

### 查看当前级别

```bash
bws config get log-level
```

### 级别选择建议

| 场景 | 建议级别 | 说明 |
|------|----------|------|
| 日常使用 | `info` | 默认级别，输出关键信息 |
| 排查问题 | `debug` | 查看详细的操作过程 |
| 深度调试 | `trace` | 开发人员使用 |
| 只看错误 | `error` | 只关注错误信息 |
| 静默模式 | `fatal` | 几乎不输出任何内容 |

### 示例

#### 查看调试信息

当遇到问题时，将日志级别调整为 debug 可以获得更多信息：

```bash
# 设置为 debug 级别
bws config set log-level debug

# 重新执行有问题的命令
bws install chrome@120

# 查看更多输出信息，帮助定位问题
```

#### 静默模式

在脚本中使用时，可能希望减少输出：

```bash
# 设置为 error 级别，只输出错误
bws config set log-level error

# 或者设置为 fatal，几乎不输出
bws config set log-level fatal
```

## 文件日志级别

文件日志默认使用 `DEBUG` 级别，始终记录详细信息，不受控制台日志级别配置的影响。

### 为什么文件日志固定为 DEBUG

- 便于问题排查：出问题时总能找到详细日志
- 占用空间小：文本日志压缩比很高
- 自动轮转：不用担心文件无限增大

### 查看文件日志

```bash
# 查看最新日志
tail -f bws-data/logs/bws.log

# 查看错误日志
grep ERROR bws-data/logs/bws.log

# 查看某个时间段的日志
grep "2024-01-15T10:" bws-data/logs/bws.log
```

### Windows 下查看日志

```powershell
# 查看日志文件
Get-Content bws-data\logs\bws.log -Tail 50

# 实时查看
Get-Content bws-data\logs\bws.log -Wait -Tail 50
```

## 日志与问题排查

当遇到问题时，日志是排查问题的重要依据。

### 一般排查步骤

1. **复现问题**：确保能稳定复现
2. **提高日志级别**：设置控制台为 debug 级别
3. **查看输出**：观察控制台输出中的错误信息
4. **查看文件日志**：在日志文件中搜索错误和警告
5. **定位问题**：根据日志中的源码位置和错误信息判断原因

### 常见日志信息说明

| 关键词 | 说明 |
|--------|------|
| `开始下载` | 下载操作开始 |
| `下载进度` | 下载进度更新 |
| `下载完成` | 下载成功完成 |
| `开始安装` | 安装操作开始 |
| `安装完成` | 安装成功完成 |
| `安装失败` | 安装出错，查看错误详情 |
| `无法识别` | 文件名无法自动识别 |
| `校验失败` | 文件校验和不匹配，可能已损坏 |
| `网络错误` | 网络连接问题 |
| `权限不足` | 文件读写权限问题 |

### 提交问题报告

如果需要提交 bug 报告，建议附上以下信息：

1. bws 版本（`bws version`）
2. 操作系统和版本
3. 复现步骤
4. 错误信息（控制台输出）
5. 相关日志片段（文件日志）

## 注意事项

1. **日志级别只影响控制台**：`log-level` 配置只影响控制台输出，文件日志始终是 DEBUG 级别
2. **日志自动轮转**：不用担心日志文件无限增大，系统会自动轮转和清理
3. **日志包含敏感信息**：日志中可能包含文件路径等信息，分享日志前注意脱敏
4. **性能影响**：提高日志级别（如 trace）可能会略微影响性能，调试后建议改回 info
5. **日志目录位置**：可以通过 `bws cache path` 或查看数据目录找到 logs 文件夹
