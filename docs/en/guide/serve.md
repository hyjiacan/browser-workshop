# Serve Service

The `bws sv` command can serve local browser installer packages over HTTP for other clients to download, suitable for building offline distribution services within a team.

## Overview

`bws sv` is a lightweight HTTP service with the following main features:

- **Browser version distribution**: Serve locally stored browser installer packages to clients on the LAN for download
- **Manifest management**: Automatically scan directories, identify files, generate manifests and checksums
- **Resume support**: Supports HTTP Range requests, allowing large file downloads to resume from where they left off
- **Auto sync**: Can be configured to automatically sync the latest versions from online sources to local storage
- **Web management interface**: Built-in HTML page for easy viewing and operation
- **REST API**: Provides complete API interfaces for easy integration

### Use Cases

- Corporate intranets without external internet access that need unified browser version distribution
- Teams sharing browser versions internally to speed up downloads
- Multiple machines in test environments needing the same browser version
- Managing browser versions in offline environments

## Quick Start

Serve configuration is managed through the `bws-serve.ini` file. The first time you run `bws sv`, a default configuration file is created automatically; edit it and rerun to start the service.

### Basic Usage

```bash
# 1. First run (automatically creates configuration file)
bws sv
# Output: Configuration file created: D:\bws\bws-serve.ini
# Edit the configuration file, then rerun

# 2. Start the service after editing bws-serve.ini
bws sv
```

After starting, you can visit `http://localhost:8080` in a browser to view the web interface.

### Configuration File Example

```ini
# bws sv configuration file
[serve]
host = 0.0.0.0
port = 8080
packages-dir =
bin-dir =
sync = false
sync-interval = 24h
sync-browsers =
sync-channels = stable
```

```bash
# Start the service after editing the configuration file
bws sv
```

### Command Line Parameters

`bws sv` (alias `bws server`) supports the following command line parameters:

| Parameter | Abbreviation | Default | Description |
|-----------|--------------|---------|-------------|
| `--dir` | `-d` | Program directory | Specify the base directory (parent of packages/ and bin/) |

Command line parameters override the corresponding settings in the configuration file.

### Configuration Items

| Config Item | Default | Description |
|-------------|---------|-------------|
| `host` | `0.0.0.0` | Listen host address |
| `port` | `8080` | Listen port |
| `packages-dir` | Program directory/packages | Browser installer package storage directory |
| `bin-dir` | Program directory/bin | Client binary storage directory |
| `sync` | `false` | Whether to enable automatic sync |
| `sync-interval` | `24h` | Sync interval (supports 30d, 24h, 30m format) |
| `sync-browsers` | All | List of browsers to sync, comma-separated (e.g. chrome,firefox) |
| `sync-channels` | `stable` | List of channels to sync, comma-separated (e.g. stable,beta) |

### Configuration File

Configuration is saved in the `bws-serve.ini` file in the program directory:

```ini
[serve]
host = 0.0.0.0
port = 8080
sync = true
sync-interval = 30d
sync-channels = stable
```

### Enabling Auto Sync

Edit the `bws-serve.ini` configuration file and set the following options:

```ini
[serve]
sync = true
sync-interval = 30d
sync-browsers = chrome,firefox
sync-channels = stable,beta
```

Then rerun `bws sv`.

For detailed instructions on automatic sync, please refer to the [Serve Auto Sync](./serve-sync.md) chapter.

## Directory Structure

Under the base directory, you need to create a `packages/` directory and place the browser installer packages inside.

### Full Directory Structure

```
serve-root/
├── packages/               # Installer package storage directory (required)
│   ├── Chrome_120.0.6099.109_Windows_x64.exe
│   ├── Chrome_121.0.6167.85_Windows_x64.zip
│   ├── Firefox_121.0_Windows_x64.exe
│   └── ...
├── bin/                    # Client binaries (optional)
│   ├── bws-windows-amd64.exe
│   └── bws-linux-amd64
└── .serve-cache.json       # Checksum cache (auto-generated)
```

### packages Directory

The `packages/` directory is required and stores all browser installer package files. Filenames are automatically recognized; supported formats and recognition rules are consistent with local import.

### bin Directory

The `bin/` directory is optional and used to store bws client binary files. Clients can download the bws program directly through the web page.

### .serve-cache.json

`.serve-cache.json` is a cache file automatically generated by the service, storing file checksums (XXH3) and metadata to speed up manifest generation. No manual editing is required.

## Filename Recognition Rules

The serve service automatically recognizes filenames in the `packages/` directory, extracting browser name, version number, platform, architecture, and channel information.

### Recognition Elements

| Element | Recognition Keywords | Examples |
|---------|----------------------|----------|
| **Browser name** | chrome, chromium, firefox | `chrome`, `firefox` |
| **Version number** | Numeric combinations like `x.y.z.w` | `120.0.6099.109` |
| **Architecture** | `win64`/`x64`/`amd64` or `win32`/`x86`/`386` | `x64`, `win64` |
| **Platform** | `windows`/`win`, `macos`/`mac`, `linux` | `windows`, `win` |
| **Channel** | `stable`, `beta`, `dev`, `canary`, `esr` | `stable`, `beta` |

### Recognizable Filename Examples

```
Chrome_120.0.6099.109_Windows_x64.exe
GoogleChrome_148.0.7778.167_Windows_x64_Offline.exe
firefox-115.0esr-win64.zip
Chrome_121.0.6167.85_Windows_x64.zip
chromium-85.0.4183.121-linux-x64.tar.gz
chromium-85.0.4183.121-linux-x64.tar.gz
44.0.2403.107_chrome64_stable_windows_installer.exe
```

### Unrecognizable Files

Files with unrecognizable filenames will not appear in the manifest, but can still be accessed via direct download URL.

If a filename cannot be automatically recognized, it is recommended to rename the file to include sufficient keywords, or use `bws i -f` to manually specify the version for installation.

## Client Configuration

Clients need to configure the offline source address to fetch versions from the serve service.

### Setting the Offline Source

```bash
bws cfg set source http://server:8080
```

Replace `server:8080` with the actual serve service address.

### Verifying Configuration

```bash
# View the remote version list; it should display versions from the serve service
bws ls --remote chrome

# Install a browser from the offline source
bws i chrome@120
```

### How It Works

After configuring the offline source, the client workflow is:

1. Execute `bws i` or `bws ls --remote`
2. Priority is given to fetching the version manifest from the configured offline source (serve service)
3. Download the installer package from the offline source
4. If the offline source does not have the required version, automatically fall back to the built-in online source

For detailed source priority instructions, please refer to the [Configuration Management](./config.md#data-source-and-priority) chapter.

## Running in Background

The serve command runs in the foreground by default, suitable for debugging and temporary use. For production environments, it is recommended to configure it as a system service to achieve auto-start on boot and automatic restart.

### Linux (systemd)

Create a systemd service file:

```bash
sudo tee /etc/systemd/system/bws-serve.service << 'EOF'
[Unit]
Description=BWS Browser Version Serve
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/bws sv
WorkingDirectory=/usr/local/bin
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
EOF
```

Enable and start the service:

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now bws-serve
sudo systemctl status bws-serve
```

### Windows (nssm)

It is recommended to use [nssm](https://nssm.cc/) (Non-Sucking Service Manager) to register serve as a Windows service.

```powershell
# 1. Download nssm and extract it to a PATH directory
# 2. Install the service
nssm install bws-serve

# In the popup window, configure:
#   Path:        D:\bws\bws.exe
#   Arguments:   serve
#   Startup dir: D:\bws

# 3. Start the service
nssm start bws-serve

# 4. Check status
nssm status bws-serve
```

Common nssm commands:

| Command | Description |
|---------|-------------|
| `nssm start bws-serve` | Start the service |
| `nssm stop bws-serve` | Stop the service |
| `nssm restart bws-serve` | Restart the service |
| `nssm remove bws-serve` | Uninstall the service |
| `nssm edit bws-serve` | Edit service configuration |

### Why Not Built-in Service Installation

bws chooses not to include system service installation functionality for the following reasons:

- **Cross-platform complexity**: Windows services, Linux systemd, and macOS launchd differ greatly, resulting in high maintenance costs
- **Permission issues**: Installing system services usually requires administrator/root privileges, which can raise security concerns
- **Flexibility**: Using mature tools like systemd/nssm allows users to more flexibly configure logs, resource limits, dependencies, etc.
- **Focus on core**: bws focuses on browser version management; service management is left to specialized tools

## Web Interface

After starting the serve service, visit the service address in a browser to see the web interface.

The web interface provides the following features:

- View statistics (number of cached versions, etc.)
- Sync control (manually trigger sync, view sync status)
- Download bws client binaries (if bin directory exists)
- API reference
